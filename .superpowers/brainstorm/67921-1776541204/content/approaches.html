Implementation Approach

Three ways to build the PS1 graphics pipeline. All deliver textures + CRT + vertex jitter, but differ in architecture.

A

Layered Pipeline (Recommended)

Build three independent systems that stack on top of each other:

  1. Texture generator — new textures.ts module with canvas-drawn procedural textures (wood, concrete, wallpaper, metal, fabric). Each scene calls it to get materials. Replaces all makeProp/makeWall solid colors.
  2. Vertex jitter shader — custom ShaderMaterial wrapper that snaps vertex positions to a configurable grid in screen space. Applied per-material.
  3. CRT post-processing — Three.js EffectComposer with a single custom shader pass: scanlines + vignette + chromatic aberration + film grain. Runs after the scene renders.
Pro: Each layer works independently — can test/debug textures without CRT, jitter without textures, etc.
Pro: Minimal changes to existing scene code — just swap material creation calls
Con: Vertex jitter via ShaderMaterial means rewriting material creation (can't use MeshLambertMaterial directly)
B

All-in-One Shader Pipeline

Single custom PS1Material that handles textures, vertex jitter, AND per-fragment effects (dithering, color banding) in one unified shader. CRT post-pass is still separate.

Pro: Maximum PS1 accuracy — can add affine texture warping in the same shader
Pro: Single material type for everything
Con: Harder to debug — one shader does everything
Con: More complex to write and maintain in a hackathon
C

Minimal-Touch Overlay

Keep existing materials mostly as-is. Add textures only to walls/floors (the biggest visual surfaces). Do CRT post-processing and vertex jitter as post-effects only (screen-space distortion, no per-vertex shader).

Pro: Fastest to implement — least code changes
Pro: No risk of breaking existing scene geometry
Con: Props still look like Minecraft blocks
Con: Screen-space jitter doesn't look as authentic as per-vertex
