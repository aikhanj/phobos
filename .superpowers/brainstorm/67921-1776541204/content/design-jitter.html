Design Section 2: Vertex Jitter Shader

The signature PS1 wobble — geometry shimmers as the camera moves

What It Does

The PS1's GPU had no sub-pixel precision. After transforming vertices to screen space, it snapped them to integer pixel coordinates. As the camera moved, vertices would "pop" between grid positions, making all geometry shimmer and wobble.

We replicate this with a custom vertex shader that:

  1. Transforms the vertex to clip space (normal MVP)
  2. Converts to screen-pixel coordinates
  3. Rounds to a configurable grid (e.g., every 2px at half-res = ~480×270 effective grid)
  4. Converts back to clip space

Implementation

New file: apps/web/src/game/ps1Material.ts

Exports a createPS1Material(opts) factory that wraps THREE.ShaderMaterial:

  • Custom vertex shader with grid-snap logic
  • Fragment shader passes through texture color + Lambert-style lighting via Three.js light uniforms
  • Accepts a map (texture), color (tint), jitterStrength (grid size)
  • Uses Three.js onBeforeCompile pattern OR full custom ShaderMaterial

The makeProp/makeWall helpers switch from MeshLambertMaterial to createPS1Material(). Lighting still works because we include Three.js lighting chunks in the shader.

Approach Decision

A

onBeforeCompile Injection (Recommended)

Patch MeshLambertMaterial's vertex shader at compile time. Keeps all built-in lighting, fog, and normal behavior — we just inject the grid-snap at the end of the vertex shader.

Pro: Free lighting, fog, all Three.js features still work
Pro: Less code — we only write the snap logic
Con: Fragile if Three.js updates shader chunks (unlikely during hackathon)
B

Full Custom ShaderMaterial

Write the entire vertex + fragment shader from scratch, including simplified Lambert lighting.

Pro: Full control — can also add affine texture warping in the same shader
Con: Must manually handle fog, lights, normals — lots more code
Con: Risk of lighting looking different from current game
